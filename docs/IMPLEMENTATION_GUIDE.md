## 0. プロダクト概要

**目的**: 精神科訪問看護師・老健スタッフ・介護福祉スタッフなどの非医師職種が、利用者の身体異常を「考えずに」主治医相談へつなげられるWebアプリ。

**本質**: 問診AIでも診断AIでもなく、**相談実行インフラ**。
「なんか変」→ 1分以内で相談文生成まで到達できる体験を最優先とする。

## 2. 絶対に守るプロダクト原則

### 2-1. Zero-thinking principle
ユーザーは考えない。AIが補完する。
症状分類・医学的判断・相談先選択・情報整理・文章作成を要求しない。

### 2-2. 許容する入力
見たこと、感じた違和感、わからない状態のみ。
「確認できていない」を常に許容する。

### 2-3. システムの責務
観察の聞き取り → 情報構造化 → リスク判定 → Quick Guideline → 相談先決定 → 相談文生成 → 送信支援

### 2-4. 安全要件
- 診断しない、処方しない、断定しない
- 身体要因優先、医師判断優先
- 緊急時は即共有誘導

### 2-5. 用語規則
- 「患者」ではなく **「利用者」「利用者さん」** を使用
- 「不明」ではなく **「未確認」** を使用
- 診断名を断定せず、**観察ベースの表現** を使用
  - ○「〜の様子がみられます」「〜の変化を認めています」
  - ×「〜の可能性があります」「〜が疑われます」

## 5. カテゴリ定義（ガイドライン準拠）

| ID | ラベル | 観察の根拠 |
|---|---|---|
| `usual_diff` | いつもと違う | ABCDEアプローチ全般 |
| `dyspnea` | 苦しそう | B（呼吸）: 呼吸数、SpO2、チアノーゼ |
| `edema` | むくんでいる | C（循環）: 圧痕、左右差、尿量 |
| `palpitation` | 動悸がする | C（循環）: 脈拍数・リズム、胸痛 |
| `low_energy` | 元気がない | ABCDEアプローチ + 脱水チェック |
| `poor_intake` | 食べられない | 脱水チェック: ツルゴール、口腔乾燥、わきの下 |
| `confusion` | ぼんやりしている | D（意識）: JCS簡略版、急性/慢性、発熱 |
| `fall` | 転んだ | 頭部外傷観察: 嘔吐、意識変容、骨折兆候 |
| `medication` | 薬が気になる | 精神科身体合併症 + 過量服薬リスク |
| `unknown_worry` | よくわからないが心配 | バイタルサイン全般 + 主観的緊急度 |

### Quick Guide の構造
- **checks**: ガイドライン準拠の観察項目（ABCDEアプローチベース）
- **redFlags**: 救急連絡基準（大阪府マニュアル + 茨城県マニュアル準拠）
- **reassurance**: 「確認できなくても大丈夫」系の安心メッセージ

## 6. リスク判定基準（救急連絡基準準拠）

### RED（緊急）— 救急連絡基準に該当
直ちに主治医連絡・救急要請の検討が必要。

| 兆候 | 根拠 |
|---|---|
| 意識障がい（呼びかけに反応しない） | 茨城県マニュアル 危険度01 |
| 呼吸停止・呼吸困難（会話不能） | 大阪府マニュアル 救急連絡基準 |
| 胸が締め付けられるような痛み | 大阪府マニュアル 救急連絡基準 |
| けいれん | 大阪府マニュアル 救急連絡基準 |
| 大量出血 | 大阪府マニュアル 救急連絡基準 |
| 頭部打撲後の嘔吐・意識変容 | 頭部外傷後観察注意事項 |
| SpO2 90%以下 | ABCDEアプローチ B（呼吸） |
| 過量服薬の疑い | 精神科訪問看護ガイドライン |

### ORANGE（準緊急）— 当日中の医師共有が必要
| 兆候 | 根拠 |
|---|---|
| SpO2 94%以下 | ABCDEアプローチ |
| 体温 37.5℃以上 | バイタルサイン基準 |
| 脈の不整・頻脈 | C（循環）評価 |
| 会話困難（短い言葉のみ） | B（呼吸）評価 |
| 脱水兆候 + 食事水分不可 | 脱水チェックガイド |
| 転倒後の疼痛・腫脹 | 転倒対応マニュアル |
| 意識変容（軽度） | D（意識）JCS |

### YELLOW（要観察）
軽度の変化があるが全身状態は保たれている。次回訪問時に報告。

### BLUE（低リスク）
緊急性を示す兆候なし。経過記録し定期訪問で観察継続。

## 7. 相談文生成仕様

### SBAR形式（富士宮市立病院SBARカード準拠）
- **S（Situation）**: 利用者に何が起きているか、いつから
- **B（Background）**: 訪問時の状況、普段との違い
- **A（Assessment）**: 観察した事実の要約（判断ではなく事実）
- **R（Request）**: 主治医への共有の緊急度と確認したいこと

### LLMプロンプト設計
- SYSTEM PROMPT: 用語規則（利用者・未確認・観察ベース表現）、SBAR定義、禁止事項を明記
- USER PROMPT: カテゴリ + 観察内容 + 評価 + RAGコンテキスト
- RAGコンテキスト: 最大500文字、スコア0.3以上の上位3チャンク

### 出力4種
| フィールド | 用途 |
|---|---|
| `summary` | 一行要約 |
| `sbar` | S/B/A/R形式の構造化報告 |
| `doctorMessage` | 主治医へ送る相談メッセージ |
| `handoverText` | 申し送りテキスト（短文） |

## 8. RAGシステム

### 処理フロー
```
PDF アップロード → pdf-parse でテキスト抽出
→ 400文字目標でチャンク分割（段落優先、50文字オーバーラップ）
→ OpenAI text-embedding-3-small で埋め込み（1536次元）
→ インメモリベクトルDBに保存

相談文生成時:
カテゴリ＋回答からクエリ構築
→ クエリを埋め込み → コサイン類似度で上位3件取得
→ 「参考資料（施設ガイドライン等）」としてプロンプトに注入
→ LLMが参照して相談文生成
→ UIに「参考資料（ガイドライン）」として紫色カードで表示
```

### フォールバック設計
| 状況 | 動作 |
|---|---|
| PDFなし | ragContext空 → 通常プロンプトで生成 |
| APIキーなし | ゼロベクトル → 検索スキップ → ダミー出力 |
| 検索ヒットなし | ragContext空 → 通常通り生成 |
| RAG例外発生 | catch → 警告ログ → 通常通り生成 |

```

  ↓ 「相談をはじめる」
/consult/start (カテゴリ選択)
  ↓ カテゴリタップ

  ↓ 「送信」or「保留」

```

## 10. UX要件

- テキスト入力は極力禁止、選択式中心
- 「確認できていない」「わからない」を常に選択可能
- 1画面1目的、各画面で迷わせない
- 医学用語を避ける（ラベルは日常語、詳細はガイドライン用語）
- スマホファースト、大きな文字、大きなボタン
- priority を色で明確化（RED=赤、ORANGE=橙、YELLOW=黄、BLUE=青）
- 性能目標: 初回答 < 2秒、全フロー完了 < 60秒、相談文生成 < 5秒
